1. PROJECT OVERVIEW
Build a bilingual (Hindi + English), mobile-first, WhatsApp-integrated platform for the State of Uttar Pradesh that enables end-to-end pregnancy registration, antenatal care (ANC) tracking, high-risk pregnancy (HRP) management, birth planning, closed-loop referrals, and real-time dashboard analytics.

1.1 Core Design Principles
The entire system must be architected around these six non-negotiable principles:
• Single longitudinal record: Every pregnant woman has one persistent pregnancy record that follows her across all facilities, visit types, and frontline workers. No fragmented episode records.
• Actionable risk management: When a pregnancy is flagged High-Risk, the system immediately triggers follow-ups, birth planning, referral logic, and creates accountability items for all stakeholders.
• Closed-loop referrals: A referral is only marked complete when the receiving facility is informed, transport is confirmed, the mother has arrived, and the outcome is captured. Open referrals auto-escalate.
• Mother-facing communication: The system proactively messages the woman, her family, and all involved health workers about due ANC visits, missed visits, birth plans, referral destinations, and transport support.
• Role-based accountability: Every stakeholder (ASHA, ANM, MO, facility, transport desk, call center, district team, state team) has a visible task queue of pending actions with timestamps and escalation rules.
• Interoperability first: Integrate with existing government systems (HMIS, RCH portal, ABDM, Civil Registration) using their existing identifiers. Do not create a parallel silo.

4. CORE MODULES — DETAILED SPECIFICATIONS
Module 1: Self-Registration via WhatsApp Chatbot
The chatbot is the primary registration entry point for beneficiaries.

4.1.1 Chatbot Flow (step-by-step)
• Step 1 — Language selection: Bot sends greeting in Hindi and English. User replies to choose language. Persist language preference.
• Step 2 — Intent detection: NLP detects pregnancy registration intent from natural language input.
• Step 3 — Mobile OTP verification: Bot sends OTP to the mobile number. User replies with OTP. On failure: retry up to 3 times, then offer fallback (see Step 3a).
• Step 3a — OTP fallback: If OTP fails (poor connectivity, shared number): offer ASHA/ANM-assisted registration pathway.
• Step 4 — Aadhaar eKYC (optional): Offer Aadhaar-based identity verification where available. If unavailable or declined, proceed without it. Never block registration on Aadhaar.
• Step 5 — Collect basic demographics: Name, age, husband name, full address (village/ward, block, district), mobile number (primary + alternate), last menstrual period (LMP) date.
• Step 6 — Calculate EDD: Derive Expected Delivery Date = LMP + 280 days. Confirm with user.
• Step 7 — Pregnancy ID generation: System auto-generates a unique Pregnancy ID. Preference: link to existing government ID (Aadhaar, RCH ID, ABHA ID) rather than creating a brand-new standalone ID.
• Step 8 — Geofencing auto-assignment: Based on the registered address, system queries the geofencing layer and auto-assigns the correct ASHA worker, ANM, sub-centre, CHC, and district.
• Step 9 — Stakeholder notification: Assigned ASHA and ANM are immediately notified via WhatsApp/SMS with the new registration.
• Step 10 — Confirmation to beneficiary: Bot sends pregnancy ID, assigned ASHA contact, first ANC due date, and summary of what to expect next.

4.1.2 Assisted Registration
Any frontline worker (ASHA, ANM, Anganwadi Worker, Panchayat Sahayak, Gram Pradhan, VHNSC member) can register a pregnancy on behalf of a woman using the worker app.
• Worker logs in with their role credentials.
• Fills the same demographic form on behalf of the beneficiary.
• Offline mode: If no internet, data is saved locally on the device and syncs when connectivity is restored. No data loss on sync failure.
• Worker is recorded as the registration source in the audit log.

4.2.3 ANC Alert & Reminder Workflow
• Due ANC reminder: System sends WhatsApp message to the beneficiary AND the assigned ASHA 7 days before the next_anc_due_date, then again 3 days before, then on the day.
• Missed ANC alert: If visit_date passes without a recorded visit, system triggers missed ANC alert to ASHA (Day 1), ANM (Day 3), and MO (Day 7) with escalating urgency.
• Overdue follow-up queue: Missed pregnancies appear in the ASHA and ANM task queue with due-day counter. Cleared only when visit is recorded.
• Repeated missed visits: If 2 or more consecutive ANC visits are missed, OR if any abnormal parameter is recorded, system escalates to ANM/MO directly with a flag for home visit.
• Every reminder message must include: beneficiary name, pregnancy ID, assigned facility name, due date, and ASHA contact.

Module 3: High-Risk Pregnancy (HRP) Flagging Engine
4.3.1 HRP Engine — When It Runs
• After every ANC visit record is saved.
• After every manual data update (e.g., lab result entry).
• On a nightly batch job to catch any records that slipped through.
The HRP engine evaluates the pregnancy record against all HRP condition codes and updates hrp_flag and hrp_conditions accordingly.

4.3.2 HRP Condition Code Reference
Implement every condition below as a discrete, testable rule in the HRP engine:

4.3.3 Post-Flagging Workflow (runs immediately when hrp_flag becomes true or new HRP code is added)
• 1. Update hrp_flag = true and append the triggering HRP code to hrp_conditions in the pregnancy record.
• 2. Send HRP alert via WhatsApp to beneficiary, ASHA, ANM, MO with: Pregnancy ID, beneficiary name, which HRP condition triggered, and required next action.
• 3. Add the pregnancy to the HRP monitoring queue visible at ASHA, ANM, MO, block, district, and state dashboards. It must stay there until delivery or record closure.
• 4. Trigger birth planning module pre-checks (see Module 4).
• 5. Schedule 30-day pre-delivery HRP alert (see Module 5 — this is mandatory).
• 6. Add home visit task to ASHA and ANM task queues.
?? CRITICAL: Once an HRP flag is set, it cannot be silently cleared. Only an authorised MO can mark an HRP condition as resolved, and the audit log must record who cleared it and when.

4.4.2 Birth Planning Workflow
• When gestational age reaches 28 weeks (or immediately on HRP flagging): system sends a WhatsApp message to the beneficiary with birth planning checklist and a link to the chatbot flow.

4.5.2 Acknowledgement & Escalation
• Every stakeholder must acknowledge their alert within 24 hours. Acknowledgement is recorded with timestamp and user ID.
• If any stakeholder does NOT acknowledge within 24 hours: system auto-escalates to their supervisor (e.g., ASHA's supervisor ANM, ANM's supervisor MO, MO's supervisor District CMO).

Module 8: AI Predictive Analytics & Nudges
Build an AI layer that operates continuously in the background to ensure accountability and reduce maternal risk.

4.8.1 AI Features Required
• Risk stratification score: Beyond binary HRP flagging, compute a continuous risk score (0–100) per pregnancy updated after every clinical entry. Higher score = more frequent follow-up.
• Missed visit prediction: Predict which pregnancies are likely to miss the next ANC visit based on historical patterns and alert the ASHA 3 days in advance.
• Abnormal parameter trend detection: If BP, Hb, or weight shows a worsening trend across visits even if individual readings are not yet HRP-threshold level, alert the ANM proactively.
• Escalation nudges: If an HRP acknowledgement is pending >12 hours, system auto-generates a personalised nudge message to the responsible stakeholder.
• Chatbot NLP: Parse free-text WhatsApp messages in Hindi and English. Handle spelling variations, colloquial terms, and voice transcription (where API supports it).
• Supply gap detection: Predict kit distribution bottlenecks before they occur based on ASHA task completion rates.

Module 9: Citizen Engagement
• Nutrition messages: Automated weekly WhatsApp messages with trimester-appropriate nutrition tips in Hindi/English.
• Danger sign alerts: System recognises danger sign keywords in beneficiary WhatsApp replies and immediately escalates to ASHA and ANM. Danger signs: fever >101.3°F for >24 hours, headache/blurring of vision, generalised swelling, palpitations/breathlessness at rest, abdominal pain, vaginal bleeding or watery discharge, reduced fetal movements.
• Government scheme information: Push messages about applicable schemes (Pradhan Mantri Matru Vandana Yojana, Janani Suraksha Yojana, etc.) based on beneficiary's stage.
• Multilingual voice/text interaction: WhatsApp supports voice messages; NLP pipeline must handle voice transcription + intent detection.

5.1 API Design Standards
• All APIs must be RESTful with JSON payloads.
• Authentication: JWT Bearer tokens; refresh token rotation.
• Every API endpoint must have a corresponding OpenAPI 3.0 specification.
• Rate limiting: Implement per-client rate limits with exponential backoff guidance.
• Idempotency: All write APIs must support idempotency keys to handle retry scenarios without duplicate records.
• Pagination: All list endpoints must support cursor-based pagination (not offset) for large datasets.

6. DATA PRIVACY & SECURITY
• Data ownership: All patient data is owned and controlled by the State Transformation Commission, Government of Uttar Pradesh. The technology partner has zero independent ownership rights.
• Encryption at rest: AES-256 for all PII fields (name, phone, Aadhaar, health data).
• Encryption in transit: TLS 1.2 minimum (TLS 1.3 preferred) for all API calls.
• Aadhaar compliance: Store only masked Aadhaar. Never store full 12-digit Aadhaar in any database or log.
• Consent framework: Beneficiary consent must be captured at registration before any data is stored. Consent withdrawal must immediately halt communications.
• RBAC enforcement: All API endpoints enforce role-based access at the server side. No front-end-only access control.
• Audit log: Every create, update, and delete action on any record must be logged with: timestamp, user_id, role, action type, and old vs. new values. Audit logs are immutable.
• PII minimisation: Logs must NOT contain PII. Use pregnancy_id and user_id as references in logs.
• Penetration testing: Platform must pass a government-approved penetration test before go-live.

8.3 GDM Thresholds
• Test: 75g oral glucose, 2-hour plasma glucose (OGTT).
• Diagnostic: 2hr PG ? 140 mg/dl = GDM (HRP-008).
• Test schedule: At first ANC contact (as early as possible). Repeat at 24–28 weeks if first test negative. Minimum 4-week gap between tests.
• If patient presents >28 weeks: one test only at first contact.

8.5 Danger Signs — Chatbot Keyword Detection Required
When any of these keywords or phrases are detected in a beneficiary's WhatsApp message, immediately alert ASHA and ANM and add to emergency queue:
• Fever (bukhar) lasting more than 24 hours
• Headache (sir dard), blurring of vision (aankhon ka dhundhla dikhai dena)
• Generalised swelling of body or face (poore shareer mein sujan)
• Palpitations, easy fatigability, breathlessness at rest (saans lene mein takleef)
• Pain in abdomen (pet mein dard)
• Vaginal bleeding or watery discharge (khoon aana ya paani aana)
• Reduced fetal movements (bacche ka hilna-doolna kam hona)
